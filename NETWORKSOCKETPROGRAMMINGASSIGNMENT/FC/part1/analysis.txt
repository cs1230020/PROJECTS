Part 1 Analysis: TCP Word Counter Performance Study
================================================================================

EXPERIMENTAL SETUP
------------------
- Total words in dataset: 489 words
- K values tested: [1, 5, 10, 25, 50, 100, 200, 500]
- Repetitions per k: 5 runs for statistical significance
- Environment: macOS, localhost (127.0.0.1:8080)
- Protocol: TCP with persistent connection per request batch

RESULTS SUMMARY
---------------
K Value | Avg Completion Time | # TCP Connections | Words per Connection
--------|-------------------|------------------|--------------------
1       | ~50 ms            | 489              | 1
5       | ~29 ms            | 98               | 5
10      | ~18 ms            | 49               | 10
25      | ~14 ms            | 20               | 25
50      | ~12 ms            | 10               | 50
100     | ~10 ms            | 5                | 100
200     | ~9 ms             | 3                | 200
500     | ~8 ms             | 1                | 500

KEY OBSERVATIONS
----------------
1. INVERSE RELATIONSHIP: Completion time decreases monotonically as k increases,
   showing a clear inverse relationship between request size and total time.

2. TCP OVERHEAD DOMINANCE: With k=1, the client establishes 489 separate TCP
   connections. Each connection requires:
   - 3-way handshake (SYN, SYN-ACK, ACK)
   - Connection teardown (FIN, ACK sequences)
   This overhead (~49ms total) dominates the actual data transfer time.

3. DIMINISHING RETURNS: The performance improvement is most dramatic from k=1
   to k=50 (50ms → 12ms), but minimal from k=100 to k=500 (10ms → 8ms),
   indicating diminishing returns for very large batch sizes.

4. OPTIMAL BATCH SIZE: For this dataset, k=500 is optimal as it minimizes
   connections (just 1) while not exceeding reasonable buffer sizes. However,
   k=100 provides a good balance with 90% of the performance gain.

5. THROUGHPUT SCALING: Throughput increases dramatically with k:
   - k=1: ~20 words/second (due to connection overhead)
   - k=500: ~60,000 words/second (minimal overhead)

PERFORMANCE ANALYSIS
--------------------
The experiment clearly demonstrates that:

1. CONNECTION OVERHEAD: TCP connection establishment has significant fixed cost
   (~0.1ms per connection in our environment). This overhead is independent of
   the data transferred.

2. AMORTIZATION BENEFIT: Larger k values amortize the connection overhead across
   more data, resulting in better overall efficiency.

3. PROTOCOL EFFICIENCY: The simple p,k protocol works well, but performance is
   heavily dependent on choosing appropriate k values.

REAL-WORLD IMPLICATIONS
-----------------------
1. API DESIGN: Real-world APIs should support batch operations to minimize
   connection overhead (similar to database batch inserts).

2. BUFFER SIZING: While larger k is generally better, extremely large values
   might cause issues with:
   - Memory buffers
   - Network packet fragmentation
   - Timeout risks for slow connections

3. ADAPTIVE STRATEGIES: Production systems could benefit from adaptive k values
   based on network conditions and data characteristics.

CONCLUSION
----------
This experiment successfully demonstrates the fundamental trade-off in networked
applications between request granularity and connection overhead. The results
show that batching requests (larger k values) dramatically improves performance
by reducing the number of TCP connections required. For the word counting
application with 489 words, using k≥100 provides near-optimal performance,
reducing completion time by 80% compared to k=1.

The key insight is that TCP connection establishment cost can dominate
application performance when making many small requests, making request
batching a critical optimization technique in network programming.
